# 问题诊断和处理

> 异常情况的识别、诊断和处理流程

---

## 中断恢复

### 场景

会话意外中断（网络断开、浏览器崩溃、系统重启等）

### 恢复步骤

1. **检查状态文件**：`cat .claude/state/current-change.json`
2. **读取关键信息**：lastUpdate, currentStage, currentTask, openIssues
3. **向用户汇报**：上次中断于哪个阶段，中断时间，上次操作
4. **根据用户确认恢复**：继续/重新开始/查看详情

### 恢复检查清单

- [ ] 状态文件已读取
- [ ] 向用户展示了恢复信息
- [ ] 用户已确认是否继续
- [ ] 如继续，已定位到正确的任务位置
- [ ] 已检查是否有未解决的问题

## 计划调整

### 场景

执行过程中发现计划不合理、需求变更、技术难题等

### 触发条件

- 发现设计缺陷
- 技术方案不可行
- 工作量严重低估
- 用户需求变更
- 发现更好的替代方案

### 处理流程

```
发现问题 → 停止执行 → 向用户报告 → 分析影响 → 提出调整方案 → 等待确认 → 执行调整
```

### 报告格式

```markdown
⚠️ 发现计划问题

**问题描述**：{详细描述}

**影响分析**：
- 影响的任务：任务 5, 6, 7
- 影响的模块：Auth 模块, User 模块
- 预计延期：1-2 天

**建议方案**：

### 方案 A：调整设计（推荐）
- 修改设计文档
- 重新规划任务 5-10
- 预计额外时间：1 天

### 方案 B：临时绕过
- 使用简化实现
- 后续优化
- 预计额外时间：0.5 天

请选择方案或提出其他建议。
```

### 调整流程

1. **评估影响**：哪些已完成的工作需要修改？哪些未开始的任务需要调整？
2. **确定回退点**：小调整（继续）、中等调整（回退到 Stage 3）、大调整（回退到 Stage 2）
3. **执行调整**：更新相关文档，更新状态文件，记录决策，通知用户

## 测试失败处理

### 诊断流程

```
测试失败 → 读取测试输出 → 分类失败类型 → 修复 → 重新运行
                               ↓
                    代码错误 | 测试问题 | 环境问题
```

### 失败类型判断

#### 1. 代码逻辑错误

**特征**：断言失败、逻辑条件错误、数据处理错误

**判断方法**：测试代码正确，预期结果合理，实际结果不符合预期

**处理**：分析失败原因 → 定位错误代码 → 修复代码 → 重新运行测试

#### 2. 测试代码问题

**特征**：断言条件不正确、测试数据有误、Mock 配置错误、选择器错误

**判断方法**：代码实际运行正常，测试预期不合理

**处理**：检查测试代码 → 修复测试脚本 → 重新运行

#### 3. 测试环境限制

**特征**：测试数据不足、环境配置特殊、浏览器行为差异、时序问题

**判断方法**：代码逻辑正确（代码审查通过），手动验证功能正常，但自动化测试失败

**处理**：识别环境限制 → 尝试调整测试环境 → 如无法调整，标记为"手动验证" → 记录到 TESTING_REPORT.md

### 处理决策树

```
测试失败
    ↓
代码审查：逻辑是否正确？
    ├─ 否 → 代码错误 → 修复代码
    └─ 是 → 继续判断
        ↓
手动验证：功能是否正常？
    ├─ 否 → 代码错误（隐藏bug）→ 深入调查
    └─ 是 → 继续判断
        ↓
测试审查：预期是否合理？
    ├─ 否 → 测试问题 → 修复测试
    └─ 是 → 环境问题 → 标记手动验证
```

### 测试失败记录

在 TESTING_REPORT.md 中记录：

```markdown
## 测试失败分析

### 失败 1：滚动按钮不可见

- **测试项**：3.2 - 桌面端按钮居中
- **失败原因**：测试环境限制（无滚动条）
- **代码状态**：✅ 代码逻辑正确
- **手动验证**：✅ 浏览器中验证通过
- **结论**：接受失败，标记为环境限制
- **验证方式**：部署后在实际环境验证
```

### 阻塞性 vs 非阻塞性

**阻塞性失败**（必须修复）：
- 核心功能不工作
- 数据损坏风险
- 安全漏洞
- 性能严重下降

**非阻塞性失败**（可接受）：
- 测试环境限制
- 边界情况（低概率）
- 视觉细节偏差
- 兼容性问题（非主流环境）

## Git 冲突处理

### 识别冲突

```bash
git status
```

输出：`both modified: src/components/Auth.tsx`

### 分析冲突

1. **读取冲突标记**：理解双方修改
2. **确定正确版本**：哪个更合理？能否合并？

### 解决策略

**策略 1：自动解决**（明确的情况）

适用：一方明显正确、简单格式差异、双方修改不同部分

处理：选择正确版本或合并 → 删除冲突标记 → 运行测试 → Git add + commit

**策略 2：询问用户**（不明确的情况）

适用：两个版本都合理、涉及业务逻辑决策、不确定哪个正确

报告格式：
```markdown
⚠️ Git 冲突需要人工决策

**冲突文件**：src/components/Auth.tsx

**版本 A（当前分支）**：简单直接
**版本 B（要合并的分支）**：更易扩展

**建议**：使用版本 B

请确认或提供其他建议。
```

### 验证解决

```bash
npm run lint
npm test
git diff --check
```

## 依赖问题处理

### 常见问题

**1. 依赖安装失败**

症状：`npm ERR! ERESOLVE unable to resolve dependency tree`

解决：清除缓存 → 删除 node_modules → 重新安装 → 检查 Node.js 版本

**2. 版本冲突**

症状：`npm WARN ERESOLVE overriding peer dependency`

解决：查看冲突详情 → 选择兼容版本 → 或使用 `--legacy-peer-deps`

**3. 缺少依赖**

症状：`Error: Cannot find module 'some-package'`

解决：`npm install some-package`

## 性能问题诊断

### 诊断步骤

1. **定位瓶颈**：Chrome DevTools Performance, React DevTools Profiler
2. **分析原因**：算法复杂度、不必要重渲染、大量 DOM 操作、网络请求过多
3. **优化方案**：算法优化、使用 memo/useCallback、虚拟滚动、请求合并/缓存
4. **验证改进**：重新测量，对比优化前后

## 问题升级机制

### 何时升级

- AI 无法独立解决
- 涉及架构级决策
- 需要外部资源（如云服务配置）
- 用户业务知识才能解决

### 升级流程

```markdown
🆘 需要人工协助

**问题**：{清晰描述}

**已尝试的方案**：
1. 方案 A - 结果：失败，原因：...
2. 方案 B - 结果：失败，原因：...

**需要的信息**：
- 问题 1：...
- 问题 2：...

**建议方案**（如有）：
- 方案 C：...

请提供指导或额外信息。
```
