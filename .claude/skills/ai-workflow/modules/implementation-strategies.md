# 实施策略

> 任务执行和质量管理策略

---

## 1. 渐进式实施

### 适用场景

- 任务数 >20
- 涉及多个独立模块
- 项目周期 >3 天
- 技术风险较高

### 执行方式

将任务分组，每组完成后立即验证。

```
第 1 组：核心逻辑（任务 1-6）
  ↓ 实施 → 单元测试 → 小结
第 2 组：UI 集成（任务 7-12）
  ↓ 实施 → 手动测试 → 小结
第 3 组：测试完善（任务 13-20）
  ↓ 实施 → E2E 测试 → 小结
```

### 分组原则

1. **按功能模块分组** - 相关任务一组
2. **按依赖关系分组** - 基础功能优先
3. **按风险分组** - 高风险任务优先

### 优点

- ✅ 早期发现问题，减少返工
- ✅ 频繁的进度反馈
- ✅ 降低变更风险
- ✅ 动态调整计划

## 2. 中间检查点

### 触发条件

- 完成 ≥50% 的任务
- 或完成所有 P0 任务
- 或实现了关键组件

### 检查内容

```markdown
📍 中间检查点

**进度**：15/49 任务已完成（31%）

**已完成核心功能**：
- ✅ 核心算法实现
- ✅ 主要组件完成

**下一步计划**：
- 边界情况处理
- 性能优化

**发现的问题**：
- [记录问题]

是否继续按计划执行？
```

## 3. Git 提交策略

### 提交频率

| 任务规模 | 提交频率 | 示例 |
|---------|---------|------|
| 小任务（<5 文件） | 任务完成即提交 | 修复 bug |
| 中任务（5-10 文件）| 功能点完成即提交 | 实现组件 |
| 大任务（>10 文件）| 阶段性提交 | 功能 1/3 |

### 提交消息格式

```
<type>(<scope>): <subject>

<body>

任务：#<task-id>
```

**type 类型**：
- `feat`: 新功能
- `fix`: Bug 修复
- `refactor`: 重构
- `test`: 测试
- `docs`: 文档
- `style`: 格式
- `perf`: 性能

### 提交原则

1. **小步提交** - 每个任务一个 commit
2. **有意义的消息** - 使用中文，清晰描述
3. **包含任务编号** - 便于追溯
4. **可回滚** - 每个 commit 可独立工作

## 4. 任务粒度控制

### 最佳粒度

- 每个任务 **1-4 小时** 最佳
- 太大（>4h）→ 拆分为子任务
- 太小（<30min）→ 合并相关任务

### 拆分示例

**原任务**（太大）：实现完整功能（8 小时）

**拆分后**：
1. 核心逻辑（2h）
2. 边界处理（1.5h）
3. 性能优化（2h）
4. 单元测试（1.5h）
5. 文档编写（1h）

## 5. 并行 vs 串行执行

### 串行执行（推荐默认）

按 tasks.md 顺序逐个执行。

**优点**：简单明确、易于调试、避免冲突

**适用**：任务有依赖关系

### 并行执行（特定场景）

多个独立任务同时进行。

**适用条件**：
- 任务完全独立
- 无共享状态
- 无依赖关系

**示例**：
- 前端 + 后端开发
- 功能实现 + 文档编写
- 不同模块测试

**注意**：需要使用 git worktree 或 `/superpowers:dispatching-parallel-agents`

## 6. 错误恢复策略

### 遇到阻塞时

```
遇到问题
    ↓
评估影响
    ├─ 阻塞当前任务 → 标记任务，继续其他任务
    ├─ 阻塞所有任务 → 向用户报告，等待解决
    └─ 不阻塞 → 记录问题，继续执行
```

### 问题记录格式

```markdown
⚠️ 阻塞问题记录

**任务**：#7 实现功能 X
**问题**：缺少依赖 Y
**影响**：阻塞任务 7-10
**临时方案**：使用替代方案
**长期方案**：配置依赖 Y

是否采用临时方案继续？
```

## 7. 质量 vs 速度平衡

### 快速交付模式

**适用**：MVP、原型、紧急修复

**策略**：
- 只实现核心功能
- 简化错误处理
- 最小化测试（Level 1 验收）
- 技术债记录到 TODO

### 标准交付模式

**适用**：常规功能开发

**策略**：
- 实现完整功能
- 完善错误处理
- 标准测试覆盖（Level 2 验收）
- 代码质量良好

### 高质量交付模式

**适用**：核心功能、关键系统

**策略**：
- 全面的功能覆盖
- 健壮的错误处理
- 完整测试覆盖（Level 3 验收）
- 优秀的代码质量
- 完善的文档

---

## 核心原则

1. **分组执行** - 降低风险，早期发现问题
2. **中间检查点** - 保持用户参与，及时调整
3. **小步提交** - Git 历史清晰，易于回滚
4. **合理粒度** - 任务大小适中（1-4h）
5. **质量平衡** - 根据场景选择交付模式
