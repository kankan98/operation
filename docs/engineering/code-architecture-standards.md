# 代码架构与实现标准

最后更新：2026-05-23

本文件是项目级工程约束，用来防止代码量增长后出现架构跑偏、重复实现、过度抽象、
依赖膨胀、难以阅读和难以迭代的问题。它补充 `.codex/rules/03-implementation-quality.md`，
不替代 OpenSpec、AGENTS.md 或安全规则。

## 核心原则

每次写代码前先回答一句话：

> 这个改动是否以最小、清晰、可验证的方式推进了当前 OpenSpec 或用户目标？

如果答案不明确，先暂停实现，补充设计、契约或任务说明。

工程目标按优先级排序：

1. 正确表达业务边界和用户工作流。
2. 保持模块职责单一、依赖方向清晰。
3. 让后续接数据库、AI、RAG、权限和集成时不需要推倒重来。
4. 避免为了“可能以后会用”提前造复杂抽象。
5. 用类型、schema、测试和文档降低理解成本。

## 编码前决策门

非平凡代码变更前必须完成这组检查。检查结论写入 OpenSpec design、tasks、代码注释、
PR/最终报告之一，粒度按风险缩放。

| 问题 | 通过标准 |
| --- | --- |
| 要不要做？ | 能对应用户目标、OpenSpec requirement、缺陷、验证缺口或明确技术债 |
| 能不能不做？ | 已考虑不改代码、文档说明、使用现有组件、配置调整或更小切片 |
| 有没有替代方案？ | 至少比较现有模式、标准库/框架能力、轻量实现和新增依赖/抽象 |
| 用了有什么负面影响？ | 明确维护成本、学习成本、性能/包体、权限/安全、测试成本、迁移成本 |
| 不用会怎样？ | 明确是否会阻塞当前目标、导致重复、产生风险或只是延后优化 |
| 值不值得？ | 收益大于复杂度，并且验证方式清楚 |
| 影响架构吗？ | 不跨越 UI、domain、data、AI、integration 边界；如必须跨越，先更新 OpenSpec |
| 怎么回滚？ | 能撤回、禁用、替换或保持静态 fallback |
| 怎么验证？ | 有对应 lint、typecheck、build、unit、browser、contract 或 manual check |

## 架构边界

当前项目必须保留这些层次。没有 OpenSpec 变更，不得随意新增或打穿边界。

| 层 | 负责 | 不负责 |
| --- | --- | --- |
| UI | 渲染、交互、本地状态、可访问性、响应式 | 直接访问数据库、LLM、RAG、外部搜索 |
| Domain | 直播、球拍、知识、话术、任务等业务规则 | HTTP 细节、SDK 细节、持久化细节 |
| Data | repository、schema、迁移、查询、事务 | UI 展示、prompt 拼接 |
| AI | prompt、model call、schema validation、retry、run state | UI 布局、数据库 schema 所有权 |
| Integration | 抖音、电商、公开来源、文件、通知等外部边界 | 领域规则和审核决策 |
| Contract | 未来 API/Repository/AI/RAG 输入输出和状态 | 可调用运行时代码 |

允许临时静态数据，但必须清楚标注为示例或预览，不能伪装成真实持久化。

## 模块和文件边界

文件开始同时承担以下任意两类职责时，应考虑拆分：

- 渲染 UI。
- 业务规则。
- 数据转换/归一化。
- API/HTTP 边界。
- AI prompt/model 处理。
- 持久化/查询。
- 外部平台调用。
- 大量 mock/static 数据。

拆分优先顺序：

1. 把静态数据放到 `src/lib/*` 或后续 data fixture。
2. 把可复用 UI 放到 focused component。
3. 把业务判断放到 domain/service。
4. 把 API、repository、adapter 放到未来明确边界中。

不要为了文件变短而机械拆分。拆分必须让职责更清晰、测试更容易或复用更真实。

## 抽象规则

新增 helper、hook、service、adapter、component、type 或 utility 前，必须满足至少一个条件：

- 已有真实重复，且重复逻辑会一起变化。
- 能隔离有意义的复杂度，例如状态机、schema validation、provider adapter。
- 是已接受架构边界的一部分，例如 `AiProviderPort`、`RetrievalPort`。
- 能表达稳定领域概念，例如 RacketProduct、SessionCapture、KnowledgeSource。
- 已有本地模式可遵循，新增抽象只是延续模式。

不得因为以下原因新增抽象：

- “以后可能会用”。
- 单次使用但包装成通用框架。
- 为了隐藏很短且清晰的代码。
- 为了绕过类型错误。
- 为了让 AI 生成代码看起来更完整。

## 依赖规则

新增 npm 包、运行时服务、外部 SDK 或工具前，必须先回答：

- 现有框架、标准 API 或本地 helper 是否已经足够。
- 这个依赖解决的问题是否是当前 OpenSpec 范围内的问题。
- 维护活跃度、许可证、生态稳定性、包体/运行时影响是否可接受。
- 失败时如何降级或回滚。
- 是否会把业务边界绑定到特定 vendor。
- 是否需要安全、隐私或数据出境评估。

重要依赖必须写入 OpenSpec design。简单开发工具也要在最终报告说明原因。

## 反冗余规则

避免这几类冗余：

- 同一业务概念在多个文件出现不同字段名。
- 多个组件各自实现相同状态 badge、卡片结构或错误展示。
- 同一数据 shape 在 UI、lib、contract 中手写不一致。
- 为了页面局部效果复制全局 token、颜色、动画或 layout 模式。
- 为了暂时展示而引入未来会删除的大量 mock 逻辑。

处理方式：

- 先确认是否是真重复，不要过早抽象。
- 对稳定重复提取 shared type、utility、component 或 contract。
- 对页面特有重复，保留局部结构但用清晰命名。
- 若重复来自需求不清，先补 OpenSpec 或 contract。

## 类型和数据规则

- 共享、持久化、AI 输出、API 输入输出必须有明确 type 或 schema。
- 领域命名优先使用业务词，不要把球拍、场次、知识统一叫 item/content。
- 不确定或缺来源字段必须显式表示 unknown/missing/reviewing，不得编造默认值。
- AI 输出不能覆盖人工事实；必须保留来源、prompt/run、review 状态。
- 长文本、空输入、重复型号、混合中英文名、别名冲突和 schema mismatch 必须在设计中覆盖。

## 前端实现规则

- 页面优先表达真实工作流，不做营销首页式 UI。
- 全局色彩、圆角、状态色、动效和密度通过 token 管理。
- 组件内不得散落硬编码品牌色、状态色和动画时长。
- 所有新增交互都要考虑 loading、empty、error、success、disabled、focus、移动端和桌面端。
- 文本溢出、按钮撑破、卡片互相遮挡属于阻断问题，不是视觉小问题。
- AI 建议、人工事实、公开来源和团队经验必须视觉上区分。

## 界面文案规则

用户界面不是开发文档。页面文案必须站在运营用户角度，告诉用户现在能做什么、当前
状态是什么、下一步怎么操作。不要把需求说明、开发边界、OpenSpec、内部业务逻辑、
架构说明或实现计划直接放在普通用户界面上。

禁止出现在常规用户界面的内容：

- “当前页面用于固定工作区入口和后续实现边界”这类开发目的说明。
- “需要未来 OpenSpec 变更逐项实现”这类流程说明。
- “不读取、不生成、不保存真实业务数据”这类面向开发者的非目标声明。
- “后续需要接入数据库/AI provider/RAG/Server Action”这类架构说明。
- 大段解释为什么这么设计、为什么未实现、未来会如何实现的说明文。

允许出现的用户可理解状态：

- “暂无商品记录”。
- “请先添加球拍型号”。
- “保存功能暂未开放”。
- “需要管理员权限”。
- “来源待审核”。
- “知识不足，暂不能生成建议”。

详细开发说明必须放到 README、OpenSpec、契约、路线文档或内部调试面板中。若必须在
产品里展示限制，也要转成简短、可操作的用户语言，例如把“未接入 PostgreSQL 持久化”
改为“暂不能保存记录”。

## AI 和 RAG 实现规则

未来 AI 功能必须拆开：

1. 输入校验。
2. 检索/上下文选择。
3. prompt 版本。
4. provider adapter。
5. 结构化输出 schema。
6. 结果校验。
7. 人工审核/反馈。
8. 持久化和审计。

禁止：

- UI 直接调 LLM。
- 模型直接写权威知识库。
- 未审核 web discovery 结果直接用于未来回答。
- 无评测替换 prompt、模型、检索规则或回答策略。

## 重构规则

允许重构：

- 为当前变更降低真实复杂度。
- 修正影响当前工作的边界混乱。
- 提取真实重复。
- 让测试或验证更明确。

不允许重构：

- 仅因为个人偏好。
- 与当前 OpenSpec 无关的大范围格式化。
- 未验证就重排大量代码。
- 在不理解现有行为时替换实现。

重构后必须说明行为是否变化。无行为变化也要运行足以证明无回归的检查。

## 代码审查清单

提交或完成前检查：

- 是否符合当前 OpenSpec 和目标用户工作流。
- 是否只改了相关文件。
- 是否没有引入无根据的依赖、抽象或目录。
- 是否保留了架构边界。
- 是否新增/修改了必要 type、schema、contract 或文档。
- 是否处理 empty、error、loading、long input、重复、权限和敏感数据。
- 是否验证了 lint、typecheck、build、OpenSpec、浏览器或相应测试。
- 是否有未解释的跳过验证。
- 是否使用中文 commit message。

## 何时暂停

出现以下情况必须暂停实现，先补设计或 OpenSpec：

- 需要引入数据库、登录、AI provider、RAG、外部平台或队列。
- 代码需要跨越多个层并且边界不清。
- 需求和现有 spec 冲突。
- 需要新增运行时依赖但收益和风险不清。
- 会处理敏感数据、客户数据、真实业务数据或 prompt。
- 无法设计可验证的完成标准。
