## Context

机会工作台是当前手动优先选品流程的主判断界面。生产审计中，用户添加商品并录入手动读数后，工作台候选详情能展示评分、推荐门控、关键原因、因子拆解、缺失信号和业务/市场辅助信息，但评分相关文本仍有大量后端英文字符串直出。

商品详情页已经通过 `localize-product-detail-score-explanations` 在展示层本地化了同一批机会评分因子和解释。机会工作台应复用相同思路，使用户在主工作台和详情页看到一致的中文评分依据。

## Goals / Non-Goals

**Goals:**

- 本地化机会工作台候选详情中的已知评分因子名称。
- 本地化关键原因、评分解释、推荐门控原因和推荐下一步中的常见英文句子。
- 本地化常见 raw value，例如 `monitoring_status=false` 和 `availability=in_stock`。
- 保持未知因子、未知解释、provider/source/error 等诊断标识原文可见。
- 添加回归测试，覆盖已知英文不直出和未知诊断回退。

**Non-Goals:**

- 不修改机会评分算法、权重、贡献、推荐门控、置信度或排序。
- 不修改后端 API、shared schema、数据库结构或已有生产数据。
- 不做全局 i18n 资源化重构；本轮只处理机会工作台评分相关展示。
- 不翻译 provider 名、source 名、API 错误码或需要精确排障的技术标识。

## Decisions

1. **在机会工作台前端展示层做确定性本地化。**
   - 原因：问题是中文 UI 可读性，后端评分合同应保持稳定；商品详情页已有可复制模式。
   - 替代方案：后端根据语言输出中文。该方案会让 API 和语言环境耦合，并影响 Chat/其他客户端。

2. **集中维护映射函数，覆盖候选详情中多个入口。**
   - 原因：相同英文句子会出现在关键原因、推荐门控、因子拆解和快照市场因子里；集中函数可减少遗漏和文案漂移。
   - 替代方案：在 JSX 各处直接替换。该方案短期快，但容易重复和漏改。

3. **未知内容回退原文。**
   - 原因：评分服务会继续演进，前端不应隐藏新因子或新诊断；保留原文比空白或错误翻译更可诊断。

## Risks / Trade-offs

- 后端新增英文解释不会自动中文化 -> 保留原文，并通过生产审计和测试逐步补充已知映射。
- `ProductDetail.tsx` 与 `Opportunities.tsx` 会暂时存在相似映射 -> 本轮优先小范围修正；后续若继续增加页面，可提取共享展示 helper。
- 部分推荐门控文案包含运营术语 -> 采用商家可理解中文，同时保留“评分不等于销量/利润事实”的语义。
