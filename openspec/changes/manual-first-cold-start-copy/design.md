## Context

生产环境空数据巡检发现，新增商品默认手动优先后，Dashboard、产品列表和预警中心的冷启动文案仍以“开始监控”“商品监控”“可监控对象”为主。当前页面组件通过 i18n key 渲染这些文本，因此本轮可以通过翻译资源和页面测试完成修复。

## Goals / Non-Goals

**Goals:**

- 让零商品冷启动入口统一表达“先添加商品，建立手动研究样本”。
- 在预警空态中说明预警依赖商品与读数基础，手动读数和可选监控都可以产生信号。
- 用页面测试防止旧的监控优先文案回归。
- 保持中英文资源语义一致。

**Non-Goals:**

- 不重命名全局监控能力、监控统计或已有 `isMonitoring` 字段。
- 不修改后端 API、数据库、调度器、采集队列或历史数据。
- 不调整新增商品表单默认值；该行为已由 `manual-first-product-create-defaults` 处理。
- 不改变预警生成条件或商品列表查询逻辑。

## Decisions

1. **只改 i18n 文案，不改页面组件结构。**
   - 原因：Dashboard、ProductsList、AlertsCenter 已经通过翻译 key 渲染目标文本，组件结构没有阻碍本轮目标。
   - 替代方案：新增专用组件或重构空态。该方案会扩大范围，且不能带来必要收益。

2. **保留“监控”作为可选能力，只移除冷启动默认目标语义。**
   - 原因：系统仍有监控状态、预警和自动检查能力，完全禁用或全局替换“监控”会破坏现有功能描述。
   - 替代方案：把所有页面中的“监控”统一替换为“研究”。该方案风险高，且超出本轮冷启动问题。

3. **测试同时断言新文案存在、旧文案缺席。**
   - 原因：只断言新文案可能漏掉同页残留的旧提示；缺席断言能覆盖生产巡检发现的回归点。
   - 替代方案：只依赖快照测试。快照噪声较高，不如明确断言业务语义。

## Risks / Trade-offs

- 英文文案与中文文案表达不完全逐字对应 → 通过同一语义约束和页面测试覆盖关键短语。
- 主规格中仍有旧的 monitoring system 描述 → 本轮只修正用户可见冷启动入口，后续可单独做产品定位规格归档。
- 生产数据可能已有商品导致空态不可见 → Playwright 复测同时检查当前页面副标题和可见冷启动状态；若生产不为空，则用 API 状态和目标页面交互确认不会出现旧冷启动短语。
