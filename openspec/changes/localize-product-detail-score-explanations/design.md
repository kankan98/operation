## Context

生产手动优先主链路审计中，临时商品录入手动读数后，商品详情页评分构成能展示分数、因子、贡献和缺失信号，但部分评分因子来自后端英文字符串。当前 `ProductDetail.tsx` 已有缺失信号本地化和部分诊断文本替换，可以沿用这一模式把已知因子和常见解释转成中文。

## Goals / Non-Goals

**Goals:**

- 本地化商品详情页评分构成中的已知因子名称。
- 本地化常见评分解释句，尤其是价格位置、价格趋势、评论代理、采集健康、监控状态、市场趋势缺失等生产中可见文本。
- 将布尔/库存等原始值显示为面向商家的中文值。
- 保持未知因子和未知解释原文可见，避免隐藏后端诊断。

**Non-Goals:**

- 不修改机会评分算法、权重、贡献、推荐结果或置信度计算。
- 不修改后端 API 返回结构或 shared schema。
- 不做完整 i18n 资源化重构；本轮只处理详情页评分构成。
- 不翻译 provider 名、source 名、API 错误码等需要精确诊断的技术标识。

## Decisions

1. **在前端详情页做展示层本地化。**
   - 原因：问题是中文 UI 的可读性，后端评分结果仍应保持稳定合同；前端已有本地化辅助函数，改动面小。
   - 替代方案：后端直接输出中文解释。该方案会耦合 API 和语言环境，影响 Chat/其他客户端。

2. **使用确定性映射和短语替换，不使用模糊翻译。**
   - 原因：评分解释影响决策，错误翻译比保留原文更危险。
   - 替代方案：正则泛化翻译所有英文。该方案容易误伤专业术语和未知诊断。

3. **未知内容回退原文。**
   - 原因：评分服务可能新增因子，前端不应吞掉新信号；回退原文比空白更可诊断。

## Risks / Trade-offs

- 后端新增英文解释不会自动翻译 → 保留原文并通过测试覆盖已知生产路径，后续发现再追加映射。
- 前端映射与后端文案存在同步成本 → 映射集中在 `ProductDetail.tsx`，测试直接覆盖展示结果。
- 中文测试环境默认语言可能是英文资源 → 详情页当前硬编码中文，本轮测试直接断言详情页中文展示。
